# Pre-Sales Brand Kit (presales-brandkit)

The single source of truth for visual identity across all pre-sales
and case-study deliverables. Every downstream skill that produces a
visual or written output — one-pagers, case study tiles, proposal
documents — **must** invoke presales-brandkit as the foundational
layer. No downstream skill re-defines colours, type, logo rules, or
tone; they compose inside them.

## Core Principle

**Identity is non-negotiable; decoration is flexible.** Colours,
typography, logo, voice, and consent rules never bend. Layout
patterns, section ordering, decorative elements, and illustration
choices can adapt per scenario — but the brand DNA stays constant
across every deliverable.

## Colour Palette

All generated deliverables (HTML, PNG, PDF) must use ONLY these
brand colours. Never introduce off-palette colours.

| Role              | Hex       | RGB              | Usage                                         |
|-------------------|-----------|------------------|-----------------------------------------------|
| Primary           | #004672   | (0, 70, 114)     | Headers, nav bar, CTAs, accent rules          |
| Secondary         | #0078D4   | (0, 120, 212)    | Subheadings, links, interactive accents       |
| Accent            | #00B7C3   | (0, 183, 195)    | Highlights, tags, badges, icon tints          |
| Dark Background   | #1B1B1B   | (27, 27, 27)     | Hero sections, footer, dark panels            |
| Light Background  | #F5F5F5   | (245, 245, 245)  | Content sections, alternating rows, cards     |
| White             | #FFFFFF   | (255, 255, 255)  | Body background, card surfaces                |
| Text Primary      | #333333   | (51, 51, 51)     | Body text, paragraph copy                     |
| Success / Value   | #107C10   | (16, 124, 16)    | Metric numbers, positive outcomes, value KPIs |
| Text Secondary    | #6B6B6B   | (107, 107, 107)  | Captions, labels, metadata, muted copy        |

Two tints are permitted for backgrounds and hover states only, derived
from the palette above — never as primary fills:
- **Primary tint** #E6EEF3 (light wash of Primary) — section dividers, table header rows
- **Accent tint** #E0F7F9 (light wash of Accent) — badge backgrounds, callout panels

## Typography

One stack, used everywhere. Do not mix in other typefaces.

- **Primary font stack:** `"Segoe UI", system-ui, -apple-system, "Helvetica Neue", Arial, sans-serif`
- **Headings and body share the stack** — hierarchy comes from weight
  and size, not from a second font family.
- **Monospace** (code, technology names, config snippets only):
  `"Cascadia Code", "Consolas", "SF Mono", monospace`

### Type scale

| Role            | Size (px / rem)   | Weight | Colour        | Line height |
|-----------------|-------------------|--------|---------------|-------------|
| Display / Hero  | 48–56 / 3–3.5     | 700    | White on dark | 1.1         |
| H1              | 36 / 2.25         | 700    | Primary       | 1.2         |
| H2              | 28 / 1.75         | 600    | Primary       | 1.25        |
| H3              | 22 / 1.375        | 600    | Secondary     | 1.3         |
| Body            | 16 / 1            | 400    | Text Primary  | 1.6         |
| Caption / Label | 13 / 0.8125       | 500    | Text Secondary| 1.4         |
| Metric number   | 40–48 / 2.5–3     | 700    | Success/Value | 1.0         |

- Body copy never smaller than 14px. Captions never smaller than 12px.
- Use sentence case for headings and buttons — not ALL CAPS (badges
  and tags may use uppercase at ≤12px with letter-spacing 0.05em).
- Maximum line length for paragraphs: ~70 characters for readability.

## Logo & Placement

- **Header:** logo sits top-left, clear of all other elements.
- **Footer:** an optional smaller monochrome logo may sit with the
  company name.
- **Clear space:** keep padding equal to the logo's cap-height on all
  sides — never crowd it.
- **Minimum size:** 24px height on screen; never render below this.
- **Colour rules:** full-colour logo on White or Light backgrounds;
  white/monochrome logo on Primary or Dark backgrounds. Never place a
  full-colour logo on a busy or mid-tone background.
- **Never:** stretch, skew, rotate, recolour off-palette, add shadows,
  or place the logo inside a coloured box that clashes with the page.

### Logo placeholder & fallback

Every deliverable reserves a logo slot in the header (and optionally the
footer). Fill it using this order of preference:

1. **Logo asset supplied** — place the provided logo in the slot,
   following the placement, clear-space, sizing, and colour-variant
   rules above.
2. **No asset, but slot must be visible** — drop in a clearly-marked
   placeholder `[Add company logo]` sized to the logo slot (logo
   clear-space preserved) so the user can see exactly where to insert
   it, and list it in the response's placeholder list.
3. **Text wordmark fallback (default when no asset)** — render the
   company name as a text wordmark: Primary colour `#004672`, H2 weight
   (600), single line, in the logo slot. This is a complete, ship-ready
   substitute — a deliverable using the wordmark is valid and needs no
   placeholder flag.
   - **Default brand name when none is supplied:** use the creative
     placeholder wordmark **"Nexora Digital"** — never default to a
     real company name (e.g. do not use "Rapid Circle"). Nexora Digital
     is a neutral, made-up stand-in that signals "swap in your brand
     here" without implying a specific organisation.
   - If the user supplies a company name, use it verbatim. Only fall
     back to "Nexora Digital" when no name is given anywhere in the
     brief or conversation.

Never invent, download, or generate a logo graphic. The choice is
always: real asset → visible `[Add company logo]` placeholder → text
wordmark (default name **Nexora Digital**). When in doubt, the text
wordmark is the safe default.

## Tone of Voice

The brand sounds **confident, clear, and grounded in evidence** —
never hype, never jargon for its own sake.

- **Third person by default** ("The team delivered…"), first person
  only on explicit request.
- **Plain over clever.** Explain every technical term in one line for a
  business reader. Lead with the outcome, then the mechanism.
- **Evidence over adjectives.** Replace "dramatically improved" with a
  number or a clearly-marked placeholder. Unsubstantiated superlatives
  are banned.
- **Empathetic about problems, confident about solutions** — describe
  the client's challenge without judgment, describe the result with
  quiet assurance.
- **Concise.** Every sentence earns its place. Prefer short sentences
  and active voice.

**Banned phrases:** "game-changer", "revolutionary", "cutting-edge",
"seamlessly" (as filler), "best-in-class", "world-class", "synergy",
"leverage" (as a verb), "next-generation", and any superlative not
backed by a metric.

## Layout Principles

- **Generous whitespace.** Let sections breathe; never fill every pixel.
- **Single-column, vertical scroll** for one-pagers — content flows
  top to bottom in a clear narrative order.
- **8px spacing grid.** All margins, padding, and gaps are multiples of
  8 (8/16/24/32/48/64).
- **Alternating section backgrounds** (White ↔ Light Background) to
  separate sections without hard borders.
- **Cards** carry a subtle radius (8px) and a soft shadow
  (`0 2px 8px rgba(0,0,0,0.06)`) — never heavy borders.
- **Consistent content width:** cap main content at ~1100px, centred,
  with comfortable side gutters.
- **Accent rules** (thin Primary/Accent lines) may underline section
  headers — used sparingly.
- **Responsive:** flexbox/grid that reflows to a single column on
  narrow viewports; no horizontal scroll at any width.

## Anonymisation Rules

When anonymisation is requested, the real client identity must be
**absent from every location** — title, body, footer, image alt-text,
file name, HTML comments, and document metadata.

- **Replace the client name** with an industry-anchored descriptor:
  "a leading logistics provider", "a global financial-services firm".
  Keep it consistent throughout the deliverable.
- **Strip identifying specifics** — product names, office locations,
  named systems, or figures unique enough to re-identify the client.
- **Keep the value story intact** — industry, scale band ("enterprise",
  "mid-market"), challenge, and outcomes remain; only identity is masked.
- **Consent is the gate.** If the anonymisation flag is unset and the
  deliverable is external-facing, default to anonymised and flag it for
  the user to confirm. Never expose a named client without explicit
  go-ahead.
- **Verify before ship** — scan the final output (including file name
  and comments) for any residual occurrence of the real name.

## Contributor Display Rules

- Display as a **card grid:** initials avatar (a circle filled with the
  Primary colour, white initials) + full name + role.
- **Never expose** email addresses, phone numbers, social links,
  employee IDs, or photos pulled without consent.
- **Initials avatars only** as placeholders — do not fetch or embed
  profile photos.
- **Order:** as specified by the user, otherwise alphabetical by first
  name.
- Roles should be concise titles ("Solution Architect", "Power Platform
  Developer") — not full job descriptions.

## Accessibility Baseline

Target **WCAG 2.1 AA** on every deliverable.

- **Contrast:** body text ≥ 4.5:1 against its background; large text
  (≥24px or 19px bold) ≥ 3:1. The palette pairings below pass:
  Text Primary #333 on White/Light; White on Primary #004672 and on
  Dark #1B1B1B.
- **Never** use the Accent #00B7C3 for small text on white — it fails
  contrast. Reserve it for fills, badges, and large elements.
- **Don't rely on colour alone** to convey meaning — pair with text,
  icon, or label (e.g., a green metric also says "reduction").
- **Semantic HTML:** use real `<h1>`–`<h3>`, `<section>`, `<ul>`, and a
  logical heading order — no skipped levels.
- **Images/icons:** meaningful ones get descriptive alt-text; decorative
  ones get empty `alt=""`.
- **Tap/click targets** at least 44×44px. Visible focus state on any
  interactive element.

## Validation Checklist

Every deliverable must pass all items before it ships. Downstream skills
invoke this checklist by name.

- [ ] **Colour** — only palette colours (plus the two permitted tints)
      used; no off-palette values anywhere.
- [ ] **Typography** — single font stack; type scale and weights
      respected; no font below the minimum sizes.
- [ ] **Logo** — correct placement, clear space, minimum size, and
      on-palette colour variant for its background.
- [ ] **Tone** — third person (unless requested otherwise), no banned
      phrases, no unsubstantiated superlatives.
- [ ] **Layout** — 8px spacing grid, generous whitespace, capped content
      width, alternating section backgrounds, responsive to one column.
- [ ] **Anonymisation** (if requested) — real client name absent from
      body, title, footer, alt-text, file name, comments, and metadata.
- [ ] **Contributors** — initials avatars only; no email, phone, social,
      employee ID, or unconsented photo.
- [ ] **Accessibility** — WCAG AA contrast passes; semantic headings;
      alt-text present; focus states visible.
- [ ] **Facts** — no fabricated numbers, names, or dates; every gap is a
      clearly-marked placeholder ([Add …]).
- [ ] **Self-contained** — for HTML, all CSS inlined; no external
      stylesheet or CDN dependency; renders correctly in a browser.
